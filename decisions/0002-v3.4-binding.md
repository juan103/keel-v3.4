# ADR-0002: KEEL v3.4 binding falsifier — check_hazard_coverage

**Status:** accepted
**Date:** 2026-06-21

## Context

KEEL v3.4's central claim is that every refusal-critical hazard from the v3.3 audit has at least one wired, biting, execution-time falsifier on its silent-escape path (d≥1). This binding ADR pre-registers the behavioral contract that makes that claim mechanically checkable: a single check that fails if any refusal-critical hazard is missing a valid falsifier, is downgraded vs the pin, or sits in an impermissible red state.

The check lives in `hazard_coverage.py` (not `check_coverage.py`) — this deviation from the generic check-coverage naming is recorded in Consequences below.

## Options considered

1. Statistical falsifier over adversary-exercise pass rate — requires a live-exercise harness before Phase 0 closes; adds Phase-3 dependency to Gate 0.
2. Behavioral falsifier via `check_hazard_coverage` — mechanically checkable from Phase 0; the check bites even when the hazard list is entirely in the `pending` (red) state; defers the live-exercise layer to Phase 3 where it belongs.
3. Prose-only contract — no mechanical guard; the kit's §6 principle ("executable verification beats prose verification") rules this out.

## Decision

Use option 2: a behavioral falsifier registered to `check_hazard_coverage` in `hazard_coverage.py`, wired into `preflight.py` from Phase 0. The check is the primary falsifier for the project's claim.

```keel-binding
type = "behavioral"
id = "falsifier.primary"
behavior = "check_hazard_coverage_blocks_uncovered_refusal_critical_hazard"
test = "tests/kit/test_hazard_coverage.py::test_missing_pinned_hazard_fails"
```

## Consequences

- **Deviation (a):** `check_hazard_coverage` lives in `hazard_coverage.py`, not `check_coverage.py`. Rationale: the module was named to reflect its specific scope (hazard coverage spine) rather than the generic check-coverage pattern. All imports, tests, and preflight wiring use the actual path.
- **Deviation (b):** The `d≥1` floor decision was taken at the Atlas dry-run phase: zero-distance rows are impermissible for refusal-critical hazards outside `accepted-risk`. This floor is encoded in the matrix's `target_defense_distance` column, not in the check itself (Phase 0 baseline rows carry `target_defense_distance=1` or higher, or are `accepted-risk`).
- `check_hazard_coverage` is called from `preflight.py`'s check list; failing the check blocks all gate ratification.
- Future phases must keep `test_missing_pinned_hazard_fails` green; removing or weakening it requires an ADR.
- The `accepted-risk` escape valve for `hz-external-validation-monoculture` and `hz-substantive-judgment-residue` is noted in `goals/load-bearing-assumptions.md`; those rows are explicitly permitted under this binding without triggering the falsifier.
